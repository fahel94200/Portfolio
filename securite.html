Sécurité du Projet BTS CIEL

Toutes les mesures de sécurité implémentées pour protéger le gestionnaire de fichiers et le dashboard administrateur

Approche Sécurité

Le gestionnaire de fichiers du dashboard administrateur est la partie la plus sensible du site BTS CIEL. Il permet aux enseignants d'uploader des cours, des projets et des logiciels. Pour éviter toute exploitation malveillante, j'ai mis en place plusieurs couches de sécurité qui se complètent. Si une couche est contournée, la suivante bloque l'attaque.

Fichiers

Sécurisation du Gestionnaire de Fichiers

6 couches de protection pour l'upload et la gestion des fichiers

Protection critique

Whitelist de Dossiers

Seuls 5 dossiers sont accessibles sur le serveur. Tout accès à un autre répertoire est automatiquement bloqué. Cela empêche un attaquant de naviguer dans l'arborescence du serveur.

// Dossiers autorisés uniquement
$whitelist = [
  'cours',
  'projet',
  'logiciels',
  'images',
  'docs'
];
Protection critique

Whitelist d'Extensions par Dossier

Chaque dossier a ses extensions autorisées. Par exemple, le dossier "images" n'accepte que les fichiers .jpg, .png et .gif. Un fichier .php ou .exe ne pourra jamais être uploadé dans ce dossier.

// Extensions par dossier
'images' => ['jpg', 'png', 'gif']
'cours'  => ['pdf', 'docx', 'pptx']
'logiciels' => ['exe', 'msi', 'zip']
Protection haute

Vérification MIME Type

L'extension seule ne suffit pas : un attaquant peut renommer un fichier malveillant "virus.php" en "virus.jpg". La vérification MIME analyse le vrai contenu du fichier (ses octets) pour confirmer qu'il correspond bien à l'extension déclarée. Un script déguisé en image est immédiatement rejeté.

// Vérifie le vrai contenu, pas juste le nom
$mime = finfo_file($finfo, $file);
// "virus.php" renommé en "virus.jpg"
// → MIME détecte "text/x-php" → BLOQUÉ
Protection haute

Anti Path Traversal

Les attaques par path traversal utilisent des séquences comme ../ pour remonter dans l'arborescence du serveur et accéder à des fichiers système sensibles (ex: ../../etc/passwd). Toute requête contenant ../ ou ..\ est bloquée.

// Requête malveillante :
"../../etc/passwd"
// → Détection de "../" → BLOQUÉ ❌
Protection moyenne

Sanitization des Noms de Fichiers

Le nom de chaque fichier uploadé est nettoyé automatiquement pour supprimer tous les caractères dangereux : espaces, caractères spéciaux, accents, points multiples. Cela empêche les injections via le nom de fichier et les problèmes d'encodage sur le serveur.

// Avant : "Mon fichier (2)..php.jpg"
// Après : "mon-fichier-2.jpg"
Protection moyenne

Double Vérification avec realpath()

Même après toutes les vérifications précédentes, la fonction realpath() de PHP est utilisée comme dernière ligne de défense. Elle résout le chemin absolu réel du fichier et confirme qu'il se trouve bien dans le dossier autorisé. Si le fichier a été déplacé ou si un lien symbolique malveillant existe, c'est bloqué.

// Vérifie le chemin RÉEL sur le disque
$realPath = realpath($path);
// Confirme qu'il est dans le bon dossier
str_starts_with($realPath, $allowedDir);
Application

Sécurité Globale de l'Application

Protections transversales appliquées à l'ensemble du site

Protection critique

Protection CSRF

Chaque formulaire du site contient un token CSRF unique généré par Laravel. Ce token est vérifié côté serveur avant l'exécution de chaque requête POST. Un attaquant qui tenterait de soumettre un formulaire depuis un site externe serait bloqué car il ne possède pas le token valide.

<!-- Token unique dans chaque formulaire -->
@csrf
// Génère : <input type="hidden"
//  name="_token" value="aB3x...">
Protection critique

Hashage des Mots de Passe (bcrypt)

Les mots de passe ne sont jamais stockés en clair dans la base de données. Ils sont hashés avec l'algorithme bcrypt fourni par Laravel. Même si la base est compromise, les mots de passe restent illisibles. Bcrypt inclut un "salt" unique pour chaque mot de passe, rendant les attaques par dictionnaire inefficaces.

// Mot de passe en clair : "monMotDePasse"
// Stocké en BDD :
"$2y$10$Xk9fJ...92 caractères..."
Protection haute

Protection XSS (Cross-Site Scripting)

Le moteur de template Blade de Laravel protège automatiquement contre les attaques XSS. Quand on affiche une variable avec la syntaxe {{ }}, Blade échappe automatiquement tout le HTML et le JavaScript. Un script malveillant injecté par un utilisateur sera affiché comme du texte brut au lieu d'être exécuté.

// Input malveillant :
"<script>alert('hack')</script>"
// Blade affiche en texte brut :
"&lt;script&gt;alert('hack')&lt;/script&gt;"
// → Le script n'est PAS exécuté ✅
Protection haute

Middleware d'Authentification (Anti Brute Force)

Le dashboard administrateur est protégé par un middleware auth qui vérifie que l'utilisateur est connecté avant chaque requête. Sans authentification, impossible d'accéder aux pages d'administration. Seuls les enseignants possédant un compte ont accès au dashboard. Cela protège aussi contre les attaques par force brute car l'accès est conditionné à une session valide.

// Routes protégées par middleware
Route::middleware('auth')
  ->group(function() {
    // Dashboard, fichiers, actualités...
  });
Protection standard

Limite de Taille (50 Mo)

Chaque fichier uploadé est limité à 50 Mo maximum. Cela empêche les attaques par déni de service (DoS) où un attaquant tenterait de saturer l'espace disque du serveur en uploadant des fichiers très volumineux de manière répétée.

Choix fonctionnel

Fichiers .exe Autorisés (Logiciels)

Les fichiers .exe sont autorisés uniquement dans le dossier "logiciels" car les enseignants ont besoin de partager des outils pédagogiques avec les élèves. Ce risque est acceptable car seuls les enseignants authentifiés via le dashboard ont la possibilité d'uploader des fichiers — les visiteurs n'ont aucun accès à l'upload.

Traçabilité

Logs d'Audit

Chaque action effectuée dans le dashboard administrateur est enregistrée dans un log d'audit. Cela permet de savoir exactement qui a fait quoi, quand et depuis quelle adresse IP. En cas d'incident, on peut retracer tout l'historique des actions.

// Informations enregistrées :
"qui"    => "admin@lycee.fr"
"quoi"   => "Upload fichier cours_php.pdf"
"quand"  => "2025-02-10 14:32:05"
"ip"     => "192.168.1.45"
"dossier" => "cours"

Les logs permettent de répondre aux questions essentielles en cas d'audit : Qui a effectué l'action ? Quoi exactement ? Quand (date et heure précises) ? Depuis quelle adresse IP ? Dans quel dossier ? C'est indispensable pour la conformité et la détection d'activités suspectes.
